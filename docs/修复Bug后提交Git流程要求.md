# AI 修复 Bug 后提交 Git 流程要求

## 一、需求目标

AI 在本地完成 Bug 修复后，不能直接提交代码。必须先检查当前 Git 仓库状态和远端状态，确认是否确实存在需要提交的问题。

核心原则：

> 本地修复 Bug 后，必须先检查 Git 状态和远端状态。只有在确认存在与本次 Bug 修复相关的有效改动时，才允许 stage、commit、pull 并 push 到远端。如果检查后没有有效改动，或者远端已经没有问题，则只告知检查结果，不要创建提交，也不要推送空内容。

---

## 二、基础流程

1. AI 先在本地根据 Bug 描述进行修复。
2. 修复完成后，运行必要的构建、检查或测试，确认本地修复没有引入新错误。
3. 检查当前 Git 工作区状态。
4. 检查当前分支与远端分支的同步状态。
5. 判断是否存在需要提交到 Git 的有效改动。
6. 如果有有效改动，则保存提交并同步到 Git。
7. 如果没有有效改动，则告知用户“本地检查无可提交内容”，不要创建提交。

---

## 三、Git 检查规则

修复完成后必须执行以下检查。

### 1. 检查当前分支

需要确认：

- 当前所在分支名称
- 当前分支是否跟踪远端分支
- 当前分支是否是本次 Bug 修复应该提交的分支

### 2. 检查工作区状态

需要检查是否存在：

- modified 文件
- added 文件
- deleted 文件
- untracked 文件

### 3. 检查差异内容

需要查看具体 diff，并判断：

- 这些 diff 是否与本次 Bug 修复相关
- 是否存在无关文件
- 是否存在临时文件、缓存文件、日志文件、构建产物

禁止提交：

- 无关改动
- 临时文件
- 缓存文件
- 日志文件
- 构建产物

### 4. 检查远端状态

需要执行远端状态检查：

1. fetch 最新远端信息。
2. 判断本地分支是否落后远端。
3. 如果远端有新提交，先 pull/rebase 或按项目规则同步最新代码。
4. 同步后再次确认本地修复仍然有效。
5. 再次检查 diff，确认提交内容仍然只包含本次 Bug 修复相关改动。

---

## 四、有有效改动时的处理

如果本地确实存在与 Bug 修复相关的有效改动，则执行以下流程：

1. 保存相关文件。
2. 只 stage 与本次 Bug 修复相关的文件。
3. 不要把无关改动加入提交。
4. 编写清晰的 commit message。
5. 提交到当前工作分支。
6. pull 最新远端内容，解决可能的冲突。
7. 如果无冲突，push 到远端分支。
8. 提交完成后，告知用户以下内容：
   - 修复了什么问题
   - 修改了哪些文件
   - commit 信息
   - 当前分支
   - 是否已经同步到远端

---

## 五、没有有效改动时的处理

如果检查后发现以下任意情况：

1. 没有本地改动；
2. 改动与本次 Bug 无关；
3. 远端已经包含了相同修复；
4. 本地修复后没有产生有效 diff；
5. 构建或检查后确认不需要改代码；

则不要提交任何内容。

此时只需要告知用户：

> 已检查 Git 状态，当前没有需要提交的有效改动，因此没有创建提交。

禁止为了完成流程而创建空提交。

---

## 六、冲突处理规则

如果 pull 过程中出现冲突：

1. 不要强行覆盖远端代码。
2. 不要自动删除他人改动。
3. 先暂停提交/同步流程。
4. 告知用户冲突文件和冲突原因。
5. 等待用户确认后再继续处理。

---

## 七、禁止行为

禁止以下行为：

1. 未检查 diff 就直接提交。
2. 提交全部文件。
3. 提交无关文件。
4. 提交日志、缓存、临时文件、构建产物。
5. 创建空提交。
6. 本地没有有效改动时仍然提交。
7. 强制 push。
8. 自动覆盖远端代码。
9. 未 pull/fetch 就直接认为远端无问题。
10. 不说明提交内容就直接推送。
11. 未确认当前分支就提交。
12. 未确认远端同步状态就提交。
13. 将格式化、配置、临时生成内容混入 Bug 修复提交。

---

## 八、最终判断标准

只有当同时满足以下条件时，才允许提交：

1. 本地存在有效代码改动。
2. 改动与本次 Bug 修复相关。
3. 本地构建或基础检查通过。
4. Git diff 已确认。
5. 当前分支正确。
6. 远端状态已检查。
7. 没有未处理冲突。
8. 提交内容不包含无关文件。
9. 提交内容不是空提交。
10. 用户未要求暂停提交。

如果不满足以上条件，则不要提交，只向用户说明检查结果。

---

## 九、推荐执行顺序

推荐按以下顺序执行：

```bash
# 1. 查看当前分支和状态
git branch --show-current
git status

# 2. 拉取远端最新信息
git fetch

# 3. 查看本地与远端差异
git status
git diff

# 4. 如远端有更新，先同步
git pull --rebase

# 5. 运行项目检查或构建
# 根据项目实际命令执行，例如：
# npm run build
# npm test
# 或项目内已有构建命令

# 6. 再次检查改动
git status
git diff

# 7. 只添加与本次 Bug 修复相关的文件
git add <相关文件>

# 8. 提交
git commit -m "fix: 修复xxx问题"

# 9. 推送到远端
git push
```

如果 `git status` 显示没有有效改动，则不要执行 `git commit` 和 `git push`。

---

## 十、最终反馈格式

### 有提交时

反馈格式：

```text
已完成 Bug 修复并提交 Git。

修复内容：
- xxx

修改文件：
- xxx

当前分支：
- xxx

Commit：
- xxx

远端同步状态：
- 已 push 到远端
```

### 无需提交时

反馈格式：

```text
已检查 Git 状态，当前没有需要提交的有效改动，因此没有创建提交。
```

### 出现冲突时

反馈格式：

```text
同步远端时出现冲突，已暂停提交流程。

冲突文件：
- xxx

冲突原因：
- xxx

需要确认后再继续处理。
```
